About Design Tracker

Design Tracker is a reporting and signposting tool for design teams working on digital products and services.

Information:

Design Tracker is not a replacement for project management tools like Jira or Trello. It works alongside them — giving design teams and stakeholders a shared view of what needs to be designed, how far along it is, and where the design assets live.

It serves two roles at once:

  • Reporting tool — an at-a-glance dashboard showing the scope and progress of design delivery across a project.
  • Signposting tool — a structured record that links stakeholders and collaborators to the right assets, proposals, and specs at every stage.

How it's structured

Every digital product or service has an inherent structure. Design Tracker reflects that with a simple three-level hierarchy.

📁
Project The top-level container. Holds the project name, description, contact, team owners, and the audiences it serves.
🗂
Feature A high-level area of the product or service — think "Navigation", "Search", or "User profiles". Features group related requirements together.
📋
Requirement A specific design deliverable nested under a feature. This is where the real detail lives — described in plain, human language rather than ticket notation.

Both features and requirements can carry a status, an owner, a spec ID, and links to external assets — keeping everything traceable back to the wider project ecosystem.

The language of a requirement

Design Tracker uses a conversational, human tone. Instead of terse ticket titles, each requirement is described across a set of purposeful fields that tell the full story of a design decision — from first principles through to the finished solution.

Description
What is this requirement? What does good look like? Describe the jobs to be done — the outcome, not the implementation.
Audiences
Who is this for? Each feature and requirement can specify which audience groups it serves — for example, Members, Admins, or Super users. This makes it immediately clear who is being designed for and why, without needing to read through the full description. Audience groups are defined at the project level and can vary from project to project.
Design thinking 💡
What approach are we thinking about taking? This is the working-out — hypotheses, directions being explored, design rationale in progress. It's a live field, updated as thinking evolves.
Open questions 🤔
What are we still working through? Unanswered questions, dependencies, things we need to learn or decide before the design can be signed off. Cleared down as resolution is reached.
Why we ❤️ it
When the solution is done — what are we excited about? Why does this approach work? This field captures the rationale for the decision, making it easier to communicate and defend the design to stakeholders.
Links 🔗
Where can the proposal, prototype, or solution be viewed? Link out to Figma, Confluence, a Jira ticket, a Miro board — wherever the work lives. Supports markdown for labelled links.

Atlas component library

Every feature and requirement can specify which Atlas pattern library components are expected to be needed to build it. Atlas components are managed globally and can be linked directly, so stakeholders can follow the connection from a design requirement straight to the component specification.

This makes Design Tracker a useful bridge between design intent and the pattern library — surfacing which components are in demand across a project and helping identify where new patterns may be needed.

Getting started quickly with AI

Setting up a project from scratch can be time-consuming. Design Tracker supports an AI-assisted workflow to help teams get up and running faster.

  1. Create a new project and open the Edit form.
  2. Click Export AI schema prompt — this downloads a structured markdown prompt that describes exactly the JSON format Design Tracker expects.
  3. Share the prompt with an AI assistant (such as Claude or ChatGPT), along with any brief, brief description, or existing documentation about your project.
  4. The AI generates a complete project structure — features, requirements, owners, audiences, and statuses — in the correct format.
  5. Save the output as a .json file and use Import JSON to load it directly into your project.
  6. Review and edit each feature and requirement in the tracker to refine the detail.

Important: Importing replaces existing data

Importing a project JSON file permanently replaces all features, requirements, owners, and config for that project. A timestamped backup is downloaded automatically before any import is applied.

Teams and ownership

Each project has its own team — a list of owners who are responsible for the design work. Owners can be added, edited, and reordered within the project settings. Individual features and requirements can also carry their own owner name, enabling clear accountability at every level of the hierarchy.

Projects also define their audiences — the user groups the product or service is designed to serve. These are set up once at the project level, then referenced on individual features and requirements to make explicit who each piece of design is for. Knowing the audience of a requirement alongside its status and spec gives stakeholders a richer picture of what is being built and for whom.
